## 1. 화면 구성 (단일 화면, `src/App.jsx`)

라우팅 없이 화면 하나로 전체 흐름이 진행됩니다. 위에서 아래 순서로:

1. 제목 `🦕 디노와 함께 글쓰기`
2. 주제 입력창 (`<input>`, placeholder "글쓰기 주제를 입력하세요")
3. 글쓰기 textarea (placeholder "여기에 글을 써보세요!")
4. **글자 수 진행 바** (아래 2번 항목)
5. **도달도 게이지** — 코칭을 한 번이라도 받은 후에만 나타남 (아래 3번 항목)
6. 코칭 버튼 (라벨은 상태에 따라 바뀜)
7. 에러 메시지 (있을 때만)
8. **피드백 카드** — 코칭 결과가 있을 때만 나타남

## 2. 글자 수 진행 바

- 기준: **400자**
- `writing.length / 400 * 100`(최대 100%)만큼 초록 막대가 채워짐
- 400자 이상이면 막대 색이 진해지고(`.complete`), 옆에 `✅ 준비 완료!` 배지 표시
- **1회차 진입 조건으로만 사용**: 최초 코칭 버튼은 400자 미만이면 비활성화됨
- 2회차(재코칭) 이후에는 이 조건이 사라짐 — 글자 수가 400자 밑으로 줄어도 재제출 가능

---

## 3. 도달도(수정 루프) 게이지 — 이번 재구성의 핵심 신규 기능

- `attainment` state가 `null`이면 게이지 자체가 렌더링되지 않음(코칭 전)
- **1회차 코칭 결과**: 품질과 무관하게 무조건 **40%**로 시작
- **2회차 이후**: 서버가 돌려준 `addressed: [boolean, boolean]`(직전 보완점 2개를 각각 고쳤는지)에서 `true` 개수 × **10%**를 이전 도달도에 더함
  - 예: 40% → 보완점 2개 다 고침 → 60% → 또 2개 다 고침 → 80% → 또 → 100% → 그 이후로도 계속 증가(120%, 150%…) 가능. **상한 없음.**
- 점수 계산은 **클라이언트(React state)** 에서 수행. AI는 O/X 판단만 하고 숫자 계산은 하지 않음(계산 오류 방지 목적)
- 게이지 시각 바 자체는 100%에서 폭이 꽉 참(더 이상 넓어지지 않음)이지만, **숫자 라벨은 실제 값**(예: `도달도 120%`)을 그대로 표시
- 100% 이상 도달 시 바 색상이 진한 파랑(`.full`)으로 강조됨

**실제 확인된 동작** (Playwright E2E, 2026-07-10): 1회차 40% → 보완점 2개 모두 반영 후 2회차 정확히 60%.

## 4. AI 코칭 흐름

1. 사용자가 "디노 코칭 받기"(1회차) 또는 "다시 코칭 받기"(2회차~) 클릭
2. 버튼 라벨이 "코칭 준비 중..."으로 바뀌고 비활성화됨(로딩 중 중복 클릭 방지)
3. `getCoachingFeedback()`이 브라우저에서 **자체 서버**(`/api/coach`)로 POST 요청
4. 서버가 Google Gemini(`gemini-2.5-flash-lite`)를 호출해 결과를 그대로 반환
5. 결과 도착 시:
   - `feedback` state 갱신 → 피드백 카드 렌더링
   - `attainment` state 갱신 (위 3번 로직)
   - `lastSubmittedWriting`, `lastImprovements` state에 이번 라운드 값 저장 → 다음 라운드 비교용

### 피드백 카드 내용
- 🦕 디노의 코칭 (제목)
- 👍 잘한 점 1개
- ✏️ 보완할 점 2개

### 프롬프트 설계 (한국어, 초등학생 대상 "디노"라는 공룡 코치 페르소나)
- **1회차 프롬프트**: 주제 + 전체 글을 주고 "잘한 점 1가지 + 보완할 점 2가지(아이가 실제로 고칠 수 있는 구체적 내용)"를 요청
- **2회차 이후 프롬프트**: 고치기 전 글 + 직전 보완점 2개 + 고친 후 글을 모두 주고, ① 각 보완점을 고쳤는지 true/false 판단 → ② 새 글 기준으로 새로운 잘한 점 1개 + 보완점 2개를 요청

## 5. 서버 API — `POST /api/coach` (`api/coach.js`, Vercel 서버리스 함수)

### 요청
```
{ topic: string, writing: string, previousWriting?: string, previousImprovements?: [string, string] }
```
- `previousWriting`과 `previousImprovements`가 **둘 다** 있어야 "2회차 이후"로 판단, 하나라도 없으면 1회차 취급

### 응답
- **1회차**: `{ strength: string, improvements: [string, string] }`
- **2회차 이후**: `{ addressed: [boolean, boolean], strength: string, improvements: [string, string] }`
- **실패 시**: `{ error: string }` + 상태코드
  - `405` — GET 등 POST 아닌 요청
  - `500` — 서버에 `GEMINI_API_KEY` 환경변수 자체가 없음
  - `400` — `topic`/`writing` 누락
  - `502` — Gemini 네트워크 오류 / Gemini가 비정상 응답 / 응답에 텍스트 없음 / JSON 파싱 실패

### Gemini 호출 방식
- Google Generative Language REST API를 **SDK 없이 `fetch`로 직접 호출**
- `responseMimeType: application/json` + `responseSchema`(1회차/2회차용 스키마 분리)로 **구조화 출력을 강제** — 정규식 텍스트 파싱은 쓰지 않음
- 키는 URL 쿼리(`?key=...`)로 전달, 서버 환경변수(`GEMINI_API_KEY`)에서만 읽음 — **브라우저에는 절대 노출되지 않음**
- 서버는 완전히 무상태(stateless). DB/세션 없음. 라운드 히스토리는 브라우저 React state로만 유지되고 매 요청마다 필요한 문맥을 통째로 실어 보냄

## 6. 클라이언트 API 클라이언트 — `src/geminiCoachService.js`

- `getCoachingFeedback({ topic, writing, previousWriting, previousImprovements })` — `/api/coach`를 fetch하는 얇은 래퍼. Gemini 호출 로직은 전혀 없음(서버로 이동함)
- `CoachingError` 클래스, `CoachingErrorType`: `NETWORK`(fetch 자체 실패) / `UNKNOWN`(그 외 실패) 두 가지만 존재
  - 예전 BYOK 시절에 있던 `AUTH`(잘못된 키) 타입은 제거됨 — 사용자가 더 이상 키를 다루지 않으므로 불필요

### 화면에 노출되는 에러 메시지
- 네트워크 실패: "네트워크 오류가 발생했어요. 다시 시도해주세요."
- 그 외 모든 실패(400/402/429/500/502 등 서버가 준 구체적 이유 불문): "코칭을 받아오지 못했어요. 잠시 후 다시 시도해주세요."
- 즉 **서버 쪽 에러 메시지 본문은 클라이언트가 읽지 않고, 항상 이 두 문구 중 하나로만 통일해서 보여줌**

## 9. 자동 저장 / 새로고침 복구 (`src/sessionStorage.js`)

- localStorage 키 `dino-writing-coach:session` 하나에 세션 전체를 JSON으로 저장
- 저장 항목: `topic`, `writing`, `feedback`, `attainment`, `lastSubmittedWriting`, `lastImprovements`, `rounds`
- `rounds`는 코칭을 받을 때마다 한 항목씩 누적되는 배열 — 각 항목은 `{ writing, strength, improvements, addressed, attainmentAfter }`. 아직 화면에 노출하는 UI는 없지만(퇴고 히스토리 기능은 다음 작업), 데이터는 이미 이 시점부터 계속 쌓이는 중
- `topic`/`writing`/`feedback`/`attainment`/`lastSubmittedWriting`/`lastImprovements`/`rounds` 중 하나라도 바뀔 때마다 `useEffect`로 자동 저장(디바운스 없음)
- 앱 최초 로드 시 저장된 값이 있으면 그대로 복구 — 새로고침해도 글, 도달도, 마지막 피드백, 버튼 라벨("다시 코칭 받기" 등) 전부 유지됨
- **"새 글 시작" 버튼**: `topic`/`writing`/`rounds` 중 하나라도 내용이 있을 때만 제목 아래에 표시됨. 클릭 시 확인창(`window.confirm`) 후 localStorage 삭제 + 모든 state를 초기값으로 리셋
- localStorage를 못 쓰는 환경(프라이빗 모드 등)에서는 저장이 조용히 실패하고 앱은 계속 정상 동작(매 새로고침마다 초기화될 뿐 에러는 안 남)
- **실제 확인된 동작** (Playwright, 2026-07-10): 글쓰기 도중 새로고침 → 복구 확인. 코칭 1회차(40%) 후 새로고침 → 피드백/도달도 복구 확인. 복구된 상태에서 2회차 진행 → 60%로 정상 증가, `rounds` 배열에 2개 누적 확인. "새 글 시작" → 초기화 + 새로고침해도 다시 안 나타남 확인.

## 11. 퇴고 히스토리 — "이전 버전 다시 보기"

- 피드백 카드 아래 `rounds.length > 0`일 때만 히스토리 토글 버튼이 나타남: `이전 버전 다시 보기 (총 N회)`
- 클릭하면 지금까지의 모든 라운드를 시간순으로 펼쳐서 보여줌(기본은 접힌 상태)
- 각 라운드 카드에 표시되는 내용:
  - 라벨: 0번째는 "초안", 이후는 "N차 수정" (`roundLabel()`)
  - 그 라운드에서 제출한 글 전문 (`round.writing`, 줄바꿈 유지)
  - **1번째 이후 라운드만**: "지난 미션 반영 확인" — 직전 라운드의 보완점 2개를 이번 글에서 고쳤는지 ✅/❌로 표시(`round.addressed`를 `rounds[index-1].improvements`와 매칭)
  - 그 라운드에서 새로 받은 코칭: 👍 잘한 점 + ✏️ 보완점 2개(다음 라운드의 미션)
- 도달도 게이지 옆에 `· N회차`로 현재까지 코칭받은 횟수 상시 표시
- 별도의 새 저장 로직은 없음 — 9번 항목의 `rounds` 배열을 그대로 읽어서 렌더링만 함(서버/데이터 모델 변경 없음)
- **실제 확인된 동작** (Playwright, 2026-07-10): 3라운드 진행(도달도 40→50→70, addressed 조합 다르게) 후 히스토리를 열어 라벨·도달도·미션 체크 표시가 모두 정확히 일치함을 확인. 새로고침해도 히스토리 유지, "새 글 시작" 시 히스토리도 함께 초기화되는 것 확인.

## 12. 퇴고 히스토리 — 변경 부분 색상 강조 (diff)

- 히스토리의 각 라운드 글에서, **초안(0번째)을 제외한 모든 라운드**는 직전 라운드 글과 단어 단위(`diffWords`, `diff` 패키지)로 비교해 강조 표시됨
  - 추가된 단어: 파랑 글자 + 밑줄 (`.diff-added`)
  - 삭제된 단어: 빨강 글자 + 취소선 (`.diff-removed`)
- 색상 하나에만 의존하지 않도록 밑줄/취소선을 함께 사용 — 색맹 학생도 형태로 구분 가능
- 히스토리를 펼치면 맨 위에 범례 문구가 항상 표시됨
- 비교 대상은 **직전 라운드와만** — 최초 초안과 비교하지 않음(각 라운드는 "그 직전에서 무엇이 바뀌었는지"만 보여줌)
- 구현 전 실제 학생 글 샘플로 단어 단위 vs 글자 단위를 프로토타입 비교했고, jsdiff의 `diffWords`가 조사/어미 변화("입니다"→"예요" 등)도 어간은 건드리지 않고 정확히 짚어내는 것을 확인해 단어 단위로 채택함
- **실제 확인된 동작** (Playwright, 2026-07-10): 어미가 여러 번 바뀌는 실제 글로 테스트 — 추가/삭제 단어들이 정확히 매칭됨, 초안 항목에는 diff 클래스가 전혀 붙지 않음(원문 그대로) 확인

## 13. 알려진 제약 / 미구현 영역

- **디노 캐릭터 이미지/애니메이션 없음**: 관련 에셋(`hero.png`, `icons.svg`)은 미사용 상태로 이미 삭제됨
- **다중 화면/라우팅 없음**: 현재 단일 화면 구조
- **테스트 프레임워크 없음**: 정식 유닛/E2E 테스트 스위트가 프로젝트에 상주하지 않음. 검증은 매번 임시 스크립트(mock fetch 검증, Playwright 스크래치패드)로 하고 삭제하는 방식이 관례
- **100% 이후 특별 연출 없음**: 도달도가 100% 넘어도 숫자만 계속 올라갈 뿐 별도 축하 효과는 없음
